# 开发验证清单 — TradingJournalAnalyzer

> 新功能或修改涉及分析计算时，在 E2E 测试通过后逐项检查。

---

## 一、指标计算验证

- [ ] 每个输出的数值能否用 Excel/手工验算？（不能 → 公式可能需要审查）
- [ ] 是否使用了正确的数据类型？（pnl vs pnl_pct，绝对金额 vs 百分比）
- [ ] 聚合方式是否正确？（sum/sum vs mean(per-item)，见 Profit Capture 案例）
- [ ] 分母为零时是否有保护？（空数组、零 invested、零 MFE）
- [ ] cost_known=False 的仓位是否已排除？
- [ ] 小样本（count < 5）是否标注了"样本不足"？

## 二、金融定义验证

- [ ] 每个行为标签是否在正例/反例场景下输出正确？（参考 FINANCE_DOMAIN.md §二）
- [ ] 标签识别的是"行为"还是"结果"？(behavior vs outcome 边界)
- [ ] 是否有概念混淆？（市场环境 ≠ 交易行为，见 BULL_TREND vs TREND 案例）
- [ ] 加仓/补仓标签是否检查了持仓状态？（见 PYRAMID/AVERAGE_DOWN 案例）
- [ ] 阈值是否与 `pattern.py` 一致？（如 CHASE=MA20×1.10 + 20日高点×0.97；BREAKOUT 成交量×1.5；FOMO=当日最高×0.98）

## 三、显示格式验证

- [ ] 所有指标显示是否用了中文？英文缩写是否在括号内解释？
- [ ] 百分比类指标是否显示为百分比？（不是小数，不是绝对金额）
- [ ] 比率类指标是否有评级说明？（优秀/良好/合格/较差）
- [ ] 是否有散户能理解的解释文字？（不止显示数字，还要解释含义）
- [ ] 信息层级是否合理？（核心可见，高级折叠）

## 四、行业标准验证

- [ ] 对照 FINANCE_DOMAIN.md §五，指标格式是否与 TradesViz/Edgewonk 一致？
- [ ] 止损回测是否用了盘中数据（day low/high），而非最终 PnL？
- [ ] 止损回测是否处理了跳空低开（fill_price = min(open, stop)）？
- [ ] 归因分析是否避免了重复计算（同一仓位多次计入不同标签）？

## 五、边界与极端值

- [ ] 空数据（0笔交易）→ 是否优雅降级而非崩溃？
- [ ] 单笔交易 → 统计是否合理（胜率 0% 或 100%，PF 为 0 或 ∞）
- [ ] 全部盈利 / 全部亏损 → PF、Expectancy 是否正确？
- [ ] 极大值（单笔盈利100倍于其他）→ 是否扭曲聚合指标？
- [ ] 跨年数据 → 日期排序、持仓天数是否正确？

## 五·补、数据隔离与副作用

- [ ] "清空数据"是否走软删除（`Trade.is_deleted=True`），KPI 是否已排除 `is_deleted` 的记录？
- [ ] `cost_known=False` 的前序持仓是否已用 `valid_positions` 排除，且 `unknown_cost_count` 有提示？
- [ ] stats 端点首次调用时 `stats_snapshot` 落库是否不影响本次返回值（仅快照）？
- [ ] admin 路由是否独立鉴权、与主用户 JWT 体系隔离（不共享 token / 依赖）？
- [ ] 下载接口文件名是否已过滤 CRLF，防 header injection？

## 六、用户理解度检查

- [ ] 如果我是第一次用的散户，这个数字我能看懂吗？
- [ ] 如果看不懂，需要加什么解释文字？
- [ ] 是否有明确的"好/坏"判断？（颜色 + 评级）
- [ ] 评价文字是否会误导用户？（PF=2.0 不应该说"拖累收益"）
