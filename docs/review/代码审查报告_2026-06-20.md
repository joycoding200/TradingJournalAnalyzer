## 整体评价

项目架构清晰，6 层数据管道设计合理，金融公式定义（PF/Expectancy/Profit Capture/MaxDD）的核心实现正确，`cost_known`/`is_deleted` 数据隔离到位，止损回测的日线 low + 跳空低开处理规范。值得肯定。

但存在 **10 个阻塞级问题**，其中 2 项组合可导致"无需凭据即接管 admin 并下载全部用户真实交割单"。测试套件红色（28 failed）。金融计算引擎在 FIFO 边界处理和标签语义上有 3 处会导致盈亏算错。解析器有一处"一笔零股让整个股票组合变期货"的致命逻辑缺陷。

以下是分级问题清单，所有 🔴 项均已由审查员亲自核实源码确认。

## 🔴 阻塞项（10 项，上线前必须修复）

### 🔴-1 JWT 密钥使用硬编码默认值，可伪造任意用户/admin token
- **文件**: `backend/app/config.py:7`
- **已核实**: `secret_key: str = "change-me-in-production"`，且 `.env` 仅含 `DATABASE_URL`，未覆盖。线上运行时密钥即此已知值。
- **原因**: JWT 用此密钥签名（`jwt.py:30`）。攻击者已知默认密钥后，可本地伪造 `{"sub":"<任意user_id>","exp":...}` 得到合法 token，直接接管任意用户账户包括 admin。
- **建议**: 启动时强制校验 `assert settings.secret_key != "change-me-in-production" and len(settings.secret_key) >= 32`，否则拒绝启动；密钥从环境变量注入。

### 🔴-2 admin 鉴权与主用户 JWT 未真正隔离，与 🔴-1 联动可无凭据接管 admin
- **文件**: `backend/app/api/admin.py:27-30, 70-77`
- **已核实**: `admin_login` 调用的 `create_token(user.id)` 与普通登录完全相同，同一 `secret_key`、同一 payload 结构，无 token type/role claim。`_require_admin` 复用 `get_current_user`，仅靠 DB `is_admin` 布尔区分。
- **原因**: `CLAUDE.md` 声称"admin 路由独立鉴权"，但实际是"DB 字段级"而非"密钥/token 级"隔离。结合 🔴-1，攻击者伪造 `sub=<admin_id>` 即通吃全部 admin 路由，含 `/api/admin/download/raw/{file_id}` 批量下载全部用户真实交割单。
- **建议**: admin 用独立 secret 签发独立 token，payload 加 `"scope":"admin"` claim 并校验；或至少修 🔴-1 后在文档澄清隔离层级。

### 🔴-3 report.py 的 _load_trades 未 scope 到 raw_file_id 且未过滤软删除 — AI 报告重复计数 + 幽灵交易
- **文件**: `backend/app/api/report.py:45-58`
- **已核实**: 这是 `analysis.py:_load_trades`（已修复版本）的未同步旧副本。按 `user_id + 日期范围`加载，而非 `raw_file_id`；缺少 `Trade.is_deleted == False` 过滤。
- **原因**: 这正是 `analysis.py` 在 docstring 里明确警告并已修复的 bug——"re-upload 重叠 trade 会被重复计数"。但 `report.py` 的副本未同步。重复上传时 AI 报告交易双倍计入；用户"清空数据"（软删除）后，已删除交易仍喂给 LLM。AI 报告与 stats 端点数据不一致。
- **建议**: 删除 `report.py` 的 `_load_trades`，复用 `analysis.py` 的实现（或抽到公共模块）。这正是 💭-7 指出的 `_load_trades` 重复定义导致的 copy-paste drift。

### 🔴-4 smart.py 在循环中途翻转 is_future — 一笔零股让整个股票组合变期货
- **文件**: `backend/app/parsers/smart.py:332-333`
- **已核实**: `is_future` 本应在循环前由列级分类确定一次（line 282），但 line 332-333 基于逐行的 `qty < 100` 翻转全局标志。
- **原因**: 只要交割单中出现一笔零股（如分红产生的 50 股、碎股卖出），整个文件后续所有股票交易都被当作期货：股票代码不再 `zfill(6)`，multiplier 被设为 10，盈亏计算全部错误且不报错。这符合 `PROJECT_EXPERIENCE.md` 反复强调的核心教训："不报错，只静默算错"比"无法识别"更危险。
- **建议**: 删除 line 332-333 的中途翻转。资产类型是文件级属性，必须在循环前一次性确定。

### 🔴-5 方向判断用子串匹配"开" — 误判"卖出开仓"为买入
- **文件**: `backend/app/parsers/smart.py:322-327`
- **已核实**: `"开" in value` 同时匹配"买入开仓"和"卖出开仓"，BUY 分支先检查所以空头开仓被误判为 BUY。`else: side = "BUY"` 默认值把无法识别方向静默归为买入。
- **原因**: 对纯 A 股现货影响较小（无做空），但代码声称支持期货，期货的"卖出开仓/买入平仓"会全部反向。此外"开"/"平"是开平仓意图，"买"/"卖"是方向，两个维度不应混在一个 if 里。
- **建议**: 优先用精确集合匹配，子串匹配仅作回退；"开"/"平"与"买"/"卖"分两个维度判断；无法识别时跳过该行而非默认 BUY。

### 🔴-6 部分孤儿卖出被静默丢弃 — 总 PnL 算错（虚高）
- **文件**: `backend/app/engine/position.py:116, 136`
- **已核实**: `build()`（FIFO，实际使用方法）处理卖出时，若 sell 数量 > 所有已买入数量且已有部分匹配（`total_qty > 0 and remaining > 0`），孤儿部分被静默丢弃，不生成 `cost_known=False` 的 position。只有纯孤儿（`total_qty == 0 and remaining > 0`）才生成 orphan。
- **原因**: 用户交割单起始日前已有持仓，交割单内先买 150 股再卖 200 股，150 股正常匹配，剩余 50 股被丢弃。这 50 股卖出收入不计入任何 position，总 PnL 偏高（少算成本）。`build_grouped()` 已正确处理此场景，可参考。
- **建议**: `total_qty > 0 and remaining > 0` 时，为剩余 `remaining` 股额外生成 `cost_known=False` 的 orphan position。

### 🔴-7 部分匹配时 sell_comm 未按比例分摊 — PnL 多扣佣金
- **文件**: `backend/app/engine/position.py:139-140`
- **已核实**: `sell_comm = trade_comm`（整个 sell trade 的佣金），但 `total_buy_comm`（line 99）已按 `matched/buy_qty` 比例分摊。部分匹配的 position 多扣了未匹配部分的卖出佣金。
- **原因**: 一个 sell 只匹配了部分 buy 时，pnl 只算了匹配部分的盈亏，但卖方佣金扣了全量，PnL 偏低。
- **建议**: `sell_comm = trade_comm * (total_qty / trade.quantity) if trade.quantity > 0 else 0`。

### 🔴-8 AVERAGE_DOWN 用"最终 PnL"判断"加仓时亏损状态" — 语义错误
- **文件**: `backend/app/engine/pattern.py:199-219`
- **已核实**: `CLAUDE.md` 明确规定"AVERAGE_DOWN 必须验证亏损状态"，代码注释也写"Verify the prior position was in a loss state at time of add"，但实际用 `prev_pos.pnl < 0`（position 最终盈亏，不是加仓时的浮亏）。
- **原因**: 加仓时浮亏 5% 但最终反弹盈利 10% → 应是 AVERAGE_DOWN 但 `pnl > 0` 漏标；加仓时浮盈 3% 但最终暴跌亏损 20% → 不应是 AVERAGE_DOWN 但 `pnl < 0` 误标。
- **建议**: 用 `last_buy_date` 当天收盘价（从 market_data 获取）vs `prev_pos.avg_entry_price` 计算加仓时浮亏。fallback 路径（line 238-248）同样完全未验证亏损状态，也需修复。

### 🔴-9 文件上传无大小/类型限制，敏感数据明文落库
- **文件**: `backend/app/api/upload.py:33`；`backend/app/models/raw_file.py:20`
- **已核实**: `content = file.file.read()` 一次性读入内存无大小上限；`raw_content = Column(LargeBinary)` 明文存储用户真实交割单（含姓名/资金/持仓）。
- **原因**: DoS（上传 GB 级文件致 OOM）；任意文件类型可上传；敏感财务数据明文落库，数据库泄露即全量暴露。
- **建议**: 限制大小（如 5MB，超限返回 413）+ 扩展名白名单 + magic bytes 校验；敏感字段应用层 AES 加密落库。

### 🔴-10 全局无速率限制 — 登录/注册/admin 登录可暴力破解
- **文件**: `backend/app/api/auth.py:77`（login）、`admin.py:70`（admin_login）、`auth.py:47`（register）
- **已核实**: search 搜索 `slowapi|ratelimit|throttle|Limiter` 命中 0，全项目无限流中间件。
- **原因**: login 与 admin_login 均无限重试，攻击者可爆破 admin 密码。结合 🔴-1 风险叠加。
- **建议**: 引入 slowapi，对登录端点限 5 次/分钟/IP + 登录失败计数 + 账户级渐进延迟锁定。

## 🟡 建议项（15 项，应修复）

### 🟡-1 AI provider 无超时、无网络异常重试
- **文件**: `backend/app/ai/provider.py:33-42, 54-62, 77-86`；`validator.py:120-134`
- **原因**: 三个 provider 的 async client 均未设 timeout（OpenAI SDK 默认 600s）；`generate_with_retry` 只在校验失败时重试，`provider.generate` 抛出的网络/限流/5xx 异常无 try/except 直接传播为 500；响应解析无空值检查。
- **建议**: 设 timeout（如 60s）+ max_retries；用 try/except 包裹 generate 对可重试异常重试；解析前检查 choices/content 非空。

### 🟡-2 store_bars 逐条查重 — 严重 N+1 查询
- **文件**: `backend/app/engine/market_data.py:74-80`
- **已核实**: 对每条 bar 执行一次 `db.query(DailyBar).filter(...).first()` 查重。一次拉取 800 条日线 = 800 次 SELECT。`market_fetcher.py:122-127` 已批量查了 existing_dates，store_bars 内部又逐条重复查。
- **建议**: 接收调用方已算好的 existing_dates set 内存判断；或一次 IN 查询批量获取；或用 `INSERT ... ON CONFLICT DO NOTHING`。

### 🟡-3 gross_loss 口径不一致 — insight.py 用 `<0`，analysis.py 用 `<=0`
- **文件**: `backend/app/engine/insight.py:136, 206` vs `backend/app/api/analysis.py:224, 237`
- **已核实**: `FINANCE_DOMAIN.md` §1 规定平本交易（pnl==0）计入亏损分母，全链路口径一致。`analysis.py` 用 `p.pnl <= 0`（含平本 ✓），`insight.py` 用 `p.pnl < 0`（排除平本 ✗）。
- **建议**: `insight.py:136` 和 `206` 统一为 `p.pnl <= 0`。

### 🟡-4 stats_snapshot 首次写入后永不更新 — 数据变更后快照陈旧
- **文件**: `backend/app/api/analysis.py:288-301`
- **原因**: 快照仅在首次访问时写入。若用户随后软删除交易并重新上传，`raw_file_id` 变了但 `stats_snapshot` 仍非空 → 快照保持旧值。`list_analyses` 展示的是过期数据。
- **建议**: `raw_file_id` 变更或交易被软删除时，清空 `stats_snapshot` 使其重新计算。

### 🟡-5 report.py 生成报告时未传 market_data — 缺市场环境标签
- **文件**: `backend/app/api/report.py:116-118`
- **已核实**: 未调用 `tag_market_patterns`（需 market_data），所以 `BULL_TREND`/`BEAR_TREND`/`BREAKDOWN` 等市场环境标签在 AI 报告中完全缺失。对比 `analysis.py:369-374` 的 `_build_category_map` 完整调用了。
- **建议**: report 生成路径复用 `analysis.py` 的 `_build_category_map`，确保标签完整一致。

### 🟡-6 PYRAMID/AVERAGE_DOWN 的 first_buy_price 跨持仓周期比较
- **文件**: `backend/app/engine/pattern.py:176-177`
- **原因**: `first_buy_price` 取该 symbol 历史上第一笔买入价格，不是当前持仓周期的首次买入价。1月买→2月卖→3月买→4月加仓，代码比较 1月 vs 4月，不触发 PYRAMID，但 4月相对 3月是更高价格应标 PYRAMID。
- **建议**: `first_buy_price` 应取与 `last_buy_price` 同一持仓周期的首次买入价。

### 🟡-7 CHASE 只满足涨幅条件也打标签，与定义的 AND 关系有偏差
- **文件**: `backend/app/engine/pattern.py:419-429`
- **已核实**: `FINANCE_DOMAIN.md` 定义 CHASE 为三条件 AND，但代码只把涨幅作为必要条件，另两个条件仅决定 confidence。只满足涨幅>15% 但不满足 MA 偏离和高点邻近的 trade 也会被打 CHASE(0.5)。
- **建议**: 严格按定义三条件都满足才打 CHASE；或至少把只满足涨幅的标签改为 POSSIBLE_CHASE。

### 🟡-8 detect() 和 parse() 静默吞异常/丢行，无日志无计数
- **文件**: `backend/app/parsers/smart.py:237-238`（detect `except: return 0.0`）、`312-368`（parse `except: continue`）
- **原因**: `PROJECT_EXPERIENCE.md` Bug 5 明确记录"异常被 detect 吞掉返回 0.0"是真实踩坑。parse 逐行 `except: continue` 静默丢行无计数——若 50% 行失败，用户看到"100笔"实际有 200 笔。
- **建议**: detect 至少 `logging.exception` 记录堆栈；parse 累计跳过行数，超阈值（如 >5%）标记 warning 或抛异常。

### 🟡-9 run_analysis 未校验 raw_file 归属 — 越权/信息泄露
- **文件**: `backend/app/api/analysis.py:72-87`
- **原因**: 创建 Analysis 时直接信任 `body.raw_file_id`，不检查该 RawFile 是否属于 `current_user`。`_load_trades` 虽有 `user_id` 保护读不到他人 trades，但 `get_stats` 会按 `raw_file_id` 查 `RawFile.filename`，泄露他人上传文件名。
- **建议**: 创建前校验 `RawFile.user_id == current_user.id`。

### 🟡-10 JWT 7 天长效、无吊销机制
- **文件**: `backend/app/config.py:9`；`backend/app/auth/jwt.py:27-30`
- **原因**: token 有效期 7 天，payload 仅 `sub`+`exp`，无 `jti`/`iat`/黑名单/登出接口。token 泄露后 7 天内无法吊销。
- **建议**: 缩短 access token（15-30 分钟）+ refresh token；引入 `jti` 黑名单实现登出/吊销。

### 🟡-11 下载文件名 CRLF 过滤不完整
- **文件**: `backend/app/api/admin.py:14-16`
- **原因**: `_safe_filename` 仅过滤 CR/LF/NUL，未过滤双引号、反斜杠、ESC 等控制字符。攻击者上传 `filename="evil".html` 可逃逸引号，潜在 header injection。
- **建议**: 白名单方式 `re.sub(r'[^A-Za-z0-9._-]', '_', name)`，并使用 RFC 5987 编码 `filename*=UTF-8''<percent-encoded>`。

### 🟡-12 扩展名检查大小写敏感 — .CSV/.XLS 被拒
- **文件**: `backend/app/parsers/base.py:117-132`
- **原因**: `endswith` 区分大小写，`.CSV`/`.XLSX` 落入 `raise ValueError`，被 `detect` 吞为 0.0 → "无法识别"。Windows 用户或某些券商导出可能含大写扩展名。
- **建议**: `filename.lower().endswith(...)`；对未知扩展名可尝试内容探测。

### 🟡-13 并发锁仅进程内有效 — 多 worker 部署失效
- **文件**: `backend/app/engine/market_fetcher.py:25-26`
- **原因**: `threading.Lock` 是进程级的。uvicorn/gunicorn 多 worker 部署时，每个 worker 有独立的 `_fetching` 集合，同一 symbol 可在不同进程并行拉取。
- **建议**: 多 worker 部署改用 DB 行锁或 Redis 分布式锁；或 DB 层唯一约束 + ON CONFLICT 保证幂等。

### 🟡-14 Validator 用"全文第一个数字"校验交易次数 — 形同虚设
- **文件**: `backend/app/ai/validator.py:42-46`
- **原因**: `r"(\d+)"` 匹配报告正文中第一个数字（可能是年份"2024"、持仓天数"5"），几乎不可能等于交易次数。校验要么永远报错触发无意义重试，要么碰巧通过。
- **建议**: 用带语义上下文的正则（如 `r"交易次数[：: ]*(\d+)"`），或改为结构化 JSON 输出校验。

### 🟡-15 mootdx 客户端无重连机制
- **文件**: `backend/app/engine/market_fetcher.py:29-36`
- **原因**: TCP 连接断开后 `_CLIENT` 仍指向失效连接，后续 `client.bars()` 持续失败（被 `except: break` 吞掉返回 0 条）。服务长期运行后行情拉取静默失效。
- **建议**: 捕获连接异常后置 `_CLIENT=None` 触发重连；或定期健康检查。

## 关键风险总结

**最危险的联动风险**: 🔴-1（默认密钥）+ 🔴-2（admin/用户共享 JWT）= 攻击者无需任何凭据即可伪造 admin token，访问 `/api/admin/download/raw/{file_id}` 批量下载全部用户真实交割单。修 🔴-1 即可切断此链路。

**数据一致性风险**: 🔴-3（report.py 未 scope）导致 AI 报告与 stats 端点数据不一致，同一个用户在同一系统看到两份矛盾的盈亏数字。
